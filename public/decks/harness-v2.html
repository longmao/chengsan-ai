从「看见设计」→「理解·执行·验证·沉淀设计」= AI Native UI 交付的源头入口

一个 frame，两类交互组件，Agent 读不出。

✓ Visual 能给
顶部卡片 + 日历网格 + 事件列表 + 配色 token——像素层。
⚠ Interaction 读不出
卡片滑动消失 · 日历滚动切月 · 吸顶周条 · 日期选中/今天/范围/禁用态 · 手势 · 键盘。
✗ Spec 读不出
卡片滑失后能否恢复 · 禁用日期不可选 · 跨月选择不丢失 · 滚动后月份正确 · 范围连续。

frame 393×944 · 187 节点 · 一个 frame 含「可滑失卡片 + 日历控件」两类组件 · 静态图能看见像素，读不出行为与约束。

收紧反馈回路，前移到源头。

问题在哪里产生，反馈就尽量在哪里出现——别等它流到用户那里。

过去 · 开环

提测 → UI 同事反馈验收清单 → 开发再改

验收压到提测后，问题靠人逐条反馈。

现在 · 闭环

代码生成 → Agent 还原度验收（60–80%）→ 工程师 + 设计师补验 → 根因分析 → 沉淀负面案例 → 反哺 Agent 进化

Playwright 验收 = 把来自 UI 验收 的反馈，拉回到 开发者终端。不是多跑一个测试，而是把错误发现的位置前移。

一句话总结

这不是单纯讲「如何用 Agent 写 UI」，

而是如何把 UI 开发从
「人肉判断 + 反复返工」升级成——

Spec 驱动 + Agent 执行
+ 自动验收 + 自动修复（达到标准 or 限定 loop/时间）
+ 工程师审核 + 设计师审核
+ 失败沉淀

的 AI Native 工程闭环。
